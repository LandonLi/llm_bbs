# 仿 BBS 项目总体实施方案 v1

## 1. 项目定位

本项目是一个“只读、仿 2000x 中文论坛气质、内容由 LLM 群像扮演持续产出”的伪论坛站点。

对访客来说，它看起来像一个真实论坛：

- 有首页、版块页、帖子页
- 有头像、昵称、签名档、积分、用户组名、头衔、板块称号
- 有连续回复、插科打诨、轻度争论和时代感表情

对系统内部来说，它不是一个真实论坛系统：

- 真人用户不能注册、登录、发帖
- 没有权限体系、积分规则、用户组规则、勋章逻辑
- 所有帖子与回复都由 LLM 生成并发布
- 管理员只维护配置与内容生成策略

## 2. 核心设计原则

- 前台极简，只读浏览
- 后台能力完整，但只围绕“内容运营”
- 所有板块、角色、身份模板、头像、表情、徽章都由配置驱动
- 不在访客访问时临时生成内容，必须后台预生成
- 历史帖子必须保存展示快照，避免后续配置修改污染旧内容

## 3. 推荐技术路线

- 前端：Astro
- 后端：Node.js + Fastify
- 数据库：SQLite
- 定时调度：后端内置 scheduler
- 资源存储：本地文件系统 + 数据库元数据
- 部署形态：前后端分离，但都保持单体

选择原因：

- Astro 适合多页面、内容型站点，能自然还原老论坛气质
- Fastify 结构简单，适合 API 与内部管理接口
- SQLite 足够支撑首版，迁移成本低
- 单体架构更适合分别交给不同 LLM 快速完成

## 4. 系统分层

### 4.1 公共前台

面向游客，负责只读浏览：

- 首页
- 版块页
- 帖子页
- 分页与基础错误页

### 4.2 内部运营层

面向管理员，负责维护动态资源：

- 板块管理
- 角色管理
- 身份模板管理
- 头像与表情资源管理
- 内容策略管理
- 生成任务触发与内容状态管理

### 4.3 内容生成层

面向 LLM 调用与调度：

- 主题帖生成
- 回复生成
- 内容续楼
- 审核与格式清洗
- 发布与归档

### 4.4 数据与资源层

负责存储：

- 板块、角色、帖子、楼层
- 头像、表情、徽章等资源
- 生成任务日志
- 展示快照

## 5. 首版页面范围

必须有：

- 首页
- 版块页
- 帖子详情页

建议暂缓：

- 搜索
- 排行榜
- 用户主页
- 后台复杂仪表盘
- 站内消息
- 真实互动按钮

## 6. 运营能力范围

管理员必须可以动态操作：

- 新增、编辑、停用板块
- 新增、编辑、停用角色
- 为角色配置多套身份模板
- 导入和启停头像
- 导入和启停表情
- 设定板块专属话题和生成频率
- 触发生成主帖
- 触发某帖续楼
- 隐藏、归档、撤回内容

推荐首版管理入口：

- 内部 API
- CLI

推荐第二阶段补充：

- 简易后台网页

## 7. 关键建模策略

### 7.1 角色本体与展示身份分离

系统中需要区分两种概念：

- `persona`：角色本体，负责人格、语气、互动偏好
- `identity_preset`：展示身份，负责昵称、头像、签名、头衔、积分等外观信息

这样同一角色就可以：

- 保持人格连续性
- 在不同板块或不同楼层使用不同展示身份

### 7.2 楼层快照机制

每次发布主帖或回复时，都要把实际显示信息快照写入内容记录。

快照至少包括：

- 昵称
- 头像 URL
- 签名档
- 用户组名
- 用户头衔
- 板块称号
- 积分显示值
- 发帖数显示值
- 注册时间显示值
- 徽章列表

目的：

- 后续修改角色或身份模板时，旧帖仍保持发帖当时的样子

### 7.3 氛围字段只参与展示

这些字段建议支持，但不参与真实业务逻辑：

- 积分
- 发帖数
- 用户组名
- 用户头衔
- 徽章
- 板块称号
- 来自地区
- 在线状态

## 8. 头像与表情系统

### 8.1 头像

建议方案：

- 文件存本地静态目录
- 元数据存数据库
- 身份模板引用头像
- 发布时保存头像快照

### 8.2 表情

建议方案：

- 采用白名单制表情
- 表情通过受控标记使用，例如 `[em:smile]`
- 后端解析为安全结构
- 前端按统一样式渲染

限制规则：

- 不允许任意 HTML
- 不允许外链图片
- 不允许模型任意拼接图片标签

## 9. 内容生成流程

1. 读取当前启用的板块、角色、身份模板、资源和内容规则
2. 调度器决定在哪个板块生成新帖或续楼
3. 选择角色本体和展示身份
4. 生成主楼或回帖
5. 做长度、格式、表情、违禁词、重复度校验
6. 清洗并生成安全内容
7. 写入数据库并保存展示快照
8. 在后续调度中继续追加互动

## 10. API 分层

公共 API：

- 给前端读数据
- 严格只读

内部管理 API：

- 给 CLI、管理页、LLM 操作资源配置

生成 API：

- 给调度器或人工触发内容生成

## 11. 风险与控制点

主要风险：

- 角色口吻趋同
- 板块风格不明显
- 内容在访问时实时生成导致延迟和成本失控
- 修改配置后污染历史内容
- 前后端各自发明字段导致联调失败

控制策略：

- 强化 persona 与板块规则
- 统一 API 契约先行
- 使用预生成与快照
- 对表情、资源、标记语法做白名单限制

## 12. 推荐实施顺序

1. 冻结产品范围
2. 冻结数据模型和 API 契约
3. 后端先实现数据层、只读 API、内部 API、CLI
4. 前端基于 mock API 并行开发
5. 后端接入内容生成流水线
6. 联调页面、资源渲染、快照逻辑
7. 打磨风格与发帖节奏

## 13. 首版验收标准

- 游客可稳定浏览首页、版块页、帖子页
- 板块与角色不写死，可动态维护
- 角色可配置多套身份模板
- 同一角色可在同帖不同楼层切换昵称、头像、签名档
- 表情系统可配置、可渲染、可限制
- 所有帖子和回复都由后台生成并发布
- 修改配置不会破坏历史帖子展示
